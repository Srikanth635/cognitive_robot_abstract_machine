_krrood's Entity Query Language (EQL) is a symbolic constraint system embedded throughout pycram. It enables declarative pre/post-condition definitions on actions and lazy grounding of underspecified plan parameters at execution time._

## Purpose

EQL provides two complementary capabilities:

1. **Selective queries** — filter existing objects in a domain by symbolic constraints. Used for action pre/post-conditions and for `UnderspecifiedNode` resolution.
2. **Generative queries** — construct new instances of a class whose parameters satisfy given constraints. Used for `underspecified(X)(variable=...)` patterns (e.g. lazily grounding a navigation target from a `CostmapLocation`).

The key abstraction is the `Variable[T]`: a lazy iterable of candidate values (its *domain*) that can be composed with logical and arithmetic operators to form a `SymbolicExpression`. An expression is evaluated by calling `.evaluate()`, yielding only bindings that satisfy all constraints.

## Core types

| Type | Role |
|------|------|
| `Variable[T]` | Symbolic placeholder; wraps a domain iterable; yields values one at a time during evaluation |
| `Literal[T]` | A singleton-domain variable for a known constant |
| `Predicate` | `Symbol` subclass for boolean tests; evaluated symbolically |
| `symbolic_function` | Decorator that returns an `InstantiatedVariable` when any argument is a `Variable`; otherwise calls the real function |
| `Match[T]` | Generated by `underspecified(Constructor)` — constructs instances of `T` from variable arguments |
| `QueryBackend` | Abstraction over EQL evaluation strategies (in-memory, SQL, probabilistic) |

## Vocabulary functions (from `factories.py`)

```python
variable(type_, domain)           # Variable[T] iterating over domain
underspecified(Constructor)        # Match[T]: generates new instances
an(entity(var)).where(cond)        # Query: select vars satisfying cond
evaluate_condition(condition)      # bool: any(condition.evaluate())
and_(*conds), or_(*conds), not_()  # logical composition
for_all(var, cond), exists(var, cond)
```

## How pycram uses EQL

### 1. Action pre/post-conditions (`ActionDescription`)

Every `ActionDescription` subclass may define:
```python
@staticmethod
def pre_condition(variables, context: Context, kwargs) -> SymbolicExpression | bool:
    return and_(GripperIsFree(manipulator), reachability_validator(...))
```

`evaluate_pre_condition()` in `ActionDescription` calls:
```python
condition = self.pre_condition(self.bound_variables, self.context, self.designator_parameter)
if not evaluate_condition(condition):
    raise ConditionNotSatisfied(...)
```

**`bound_variables`** (a `@cached_property`):
```python
{field.name: variable(type(field_value), [field_value]) for field in self.fields}
```
Each concrete argument is wrapped in a single-element-domain `Variable`. This lets the condition expression treat field values symbolically even though they are already known — enabling composition with other symbolic predicates like `GripperIsFree`.

Condition evaluation is **skipped entirely** when `context.evaluate_conditions = False` (e.g. in tests).

### 2. Underspecified plan parameters (`UnderspecifiedNode` and `TransportAction`)

```python
# TransportAction: lazily ground navigation target at execution time
underspecified(NavigateAction)(
    target_location=variable(Pose, domain=CostmapLocation(...)),
    keep_joint_states=True,
)
```

`underspecified(NavigateAction)` creates a `Match[NavigateAction]`. When evaluated by the backend, `EntityQueryLanguageBackend._evaluate_underspecified`:
1. Iterates all combinations of the `variable(...)` domains.
2. For each combination, constructs a `NavigateAction(**kwargs)`.
3. Filters by any `.where()` conditions.

`UnderspecifiedNode` drives this at plan execution: `context.query_backend.evaluate(underspecified_expr)` yields grounded `ActionDescription` instances one at a time, each becoming a child `ActionNode`.

### 3. EQL inline queries in composite actions (`TransportAction.open_container`)

```python
an(entity(drawer := variable(Drawer, domain=world.semantic_annotations))
   .where(drawer.root == container)
).evaluate()
```
Finds all `Drawer` annotations attached to `container` in the world. The `:=` walrus operator binds the variable for reuse in the `.where()` clause.

### 4. `@symbolic_function` decorator (`ViewManager.get_end_effector_view`)

When called with a `Variable` argument, returns an `InstantiatedVariable` representing the call symbolically. When called with a concrete value, executes normally. This makes `ViewManager.get_end_effector_view(arm_variable, robot)` composable with `and_()` in pre-conditions.

## QueryBackend implementations

| Backend | Strategy | Used when |
|---------|----------|-----------|
| `EntityQueryLanguageBackend` | In-memory Python evaluation | Default; set in `Context.query_backend` |
| `SQLAlchemyBackend` | Translates EQL to SQL via `eql_to_sql` | When world state lives in a DB |
| `ProbabilisticBackend` | Samples from tractable probabilistic model (`FullyFactorizedRegistry`) | When parameters need stochastic grounding |

The backend is pluggable per-`Context`. Switching to `ProbabilisticBackend` in a `Context` makes all `underspecified(...)` resolutions probabilistic without changing any action code.

## Evaluation semantics

- `Variable.evaluate()` yields one `OperationResult` per domain value, carrying the binding `{variable._id_: value}`.
- Composed expressions (AND, OR, Comparator) short-circuit on their children's bindings.
- `evaluate_condition(cond)` returns `True` iff `any(cond.evaluate())` — at least one binding satisfies all constraints.
- `an(entity(v))` returns an iterable; **`evaluate_condition` is not the same as full enumeration** — it stops at the first solution.

## Open questions

- `TransportAction.open_container` evaluates an EQL query that finds `Drawer` annotations in `world.semantic_annotations`. The domain scoping (whole-world vs. subtree) isn't explicit in the query — whether it's safe to run this concurrently with world mutations is unknown.
- `ProbabilisticBackend` uses `FullyFactorizedRegistry` by default. How priors are provided (or learned) for action parameters like `Pose` is not documented in the pycram codebase.
